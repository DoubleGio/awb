Introduction

While we were creating our page we got constantly had to deal with making our page more accessible and usable. We our wanted our content to be available through multiple devices and different user groups. Our page features alts for images, different CSSs for different devices, a good text/background colour contrast, due to shadowed text and a good semantic structure. With use of sections, lists and headers. It also features a block of navigation links and easy to understand language (we avoided the use of difficult words).

Also, in terms of usability, our site has sortable tables and uses AJAX to get data, post data (from a database) and update the table with the data without having to refresh the page. It's easy to use, also for the developers, since we created a separate javascript file for the scripts we used for this. In this javascript file we also made use of JQuery, for its ease of use.

Accessibility Highlights

We found the most relevant guidelines to be:

1.4: "Make it easier for users to see and hear content including separating foreground from background."
We picked a more interesting background and to make it work, we had to carfully think about our use of colour and text style.
To make it work, we changed the background picture (of a brick wall) to be more "background friendly", by slightly dimming and blurring the picture in photoshop. As for the text, we went with a sans-serif font with a white colour. However this wasn't distinguishable enough, so we added an 1px black shadow to the text, making everything much more readable.
2.4: "Provide ways to help users navigate, find content, and determine where they are."
Our webpage provides a navigation block, allowing for quick and easy navigation to anywhere on the page. Combined with sequential and logical headers and lables makes for a smooth browser experience.

Improvement Plan

Something that clashed with some of the WCAG was the sortable table. It's a javascript file from online that we implemented without changing. To improve it we'd have to make it so you can access it using the tab keys and that you can know beforehand that the table is sortable. We tried another table, but without success. But we did manage to fix the problem of knowing beforehand, by labeling the table.

A technical implementation detail was the form, not being in the table, which caused the form to be "out of shape". To fix this we created a new seperate row for the form, it actually was really simple to do.

The colour scheme remark was made before we made the improvements to the background and text style. The background is more "background friendly" and the text has a 1px black shadow. Also, all the links work now, they just didn't have anything to link to before.

The CSS file we used also wasn't made well, the page-breaks weren't implemented correctly. We fixed this by creating a specific class called "page-break" and applied it per object to page-break-after.

Also a technical problem was the part for handheld media redefining a lot of stuff that already had been said before. We cleaned up this part and images no longer go off screen.

It was not stated clearly before that AJAX was needed to perform a reset, we fixed this now however. The reset button now makes an AJAX GET request correctly.

We renamed the javascript file to table_scripts.js
